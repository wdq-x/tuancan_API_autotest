# 章节四：如何把问题说清楚——从"感觉"到"可执行"的描述

这一章我想聊的是一个非常"术"的问题：  
**怎么把脑子里的模糊感觉，变成 AI 和同事都能执行的描述。**

## 1. 为什么"说清楚问题"这么重要？

从底层原理看，有两个关键点：

- 语言模型本质上是一个"根据上下文预测下一个词"的统计机器。  
  你给它什么样的语言，它就会往它见过的、最类似的"行业话语"上去靠。
- 现实中的需求又往往是模糊的、混杂的，里面同时掺了动机、情绪、意图和细节。

这意味着：

- **越专业、越贴近真实场景的描述，越容易命中模型在训练中见过的"模式"。**
- **越含糊、越口语化、越缺细节的描述，越容易让模型去瞎"脑补"。**

把问题说清楚，本质上是在做两件事：

1. 把你脑子里的"高维模糊图像"，压缩成一段 AI 能理解的"专业语言"。  
2. 同时把这段语言，变成团队里任何一个人看了都能执行的"任务说明书"。

## 2. 第一性原理拆解：一条高质量问题描述应该包含什么？

我自己的经验是，可以用四个关键词来检查：**对象、目标、约束、语境**。

1. **对象（Object）**：到底在说谁 / 什么？
   - 系统中的哪个模块、哪类用户、哪张表、哪个接口？
   - 避免用"这块、这里、这个逻辑"这种指代不清的表述。
2. **目标（Goal）**：希望它"变成什么样"？
   - 是要更快、更稳、更好看，还是要减少某类错误？
   - 最好能落到「可观察的变化」，而不是抽象的"优化一下体验"。
3. **约束（Constraints）**：哪些是不能动的？
   - 性能预算、兼容性要求、合规限制、现有用户习惯。
   - 这些约束往往决定了"不能怎么做"，比"怎么做"更重要。
4. **语境（Context）**：这是在什么场景下发生的？
   - 前后置条件是什么？是线上环境还是本地调试？  
   - 有没有相关的历史 bug、方案或决策可以参考？

如果一条问题描述，把这四点都说清楚了，大多数情况下，AI 和人都会"心里有数"。

## 3. 从"专业性"角度看：为什么术语和行业语言有用？

这里有一个很重要的底层逻辑：

> AI 是语言模型，都是基于行业数据训练的，所以专业的描述更符合它的"习惯"。

换个说法就是：

- 模型在训练时，见过大量"行业内的人是怎么说话的"。  
- 当你用接近行业惯例的术语和句式描述时，它更容易检索出相似的模式，给出靠谱的补全。

几个简单例子：

- 比起说"表有点乱"，用"当前订单表存在严重的反规范化，字段 A/B/C 有重复含义"更有信息量；
- 比起说"页面卡"，用"在首屏渲染时，TTI 超过 5s，主要时间耗在首个接口响应"更接近专业语境；
- 比起说"这文案太油"，用"希望语气从'促销式'调整为'中性解释 + 轻微鼓励'"更容易被模型理解。

这里不是要你"装专业"，而是尽量靠近行业内真实的表达方式，减少歧义。

## 4. 操作层面：我现在写问题描述的一个小模板

我自己日常跟 AI 或同事描述问题，会尽量用这样一个结构：

1. **背景**：一句话说清楚这是在什么系统 / 场景下发生的。
2. **当前表现**：客观描述现在发生了什么（包含必要的数据和截图信息）。
3. **期望表现**：清楚说出"理想情况下应该是什么样"。
4. **约束与顾虑**：哪些是这次不能动的，或者特别需要注意的地方。
5. **补充材料**：相关链接、接口定义、日志片段、历史讨论等。

你可以把这五条写给人看，同时也直接喂给 AI。  
这样可以最大程度保证"人和模型看到的是同一个问题"。

## 5. 一个对照示例

最后给一个简单的对照，你可以体会一下差异：

- 模糊版：
  - "帮我查一下这个页面为什么这么卡，然后优化一下。"
- 精准版：
  - "在 Chrome 最新版下，用户首次打开 `orders` 页面时，TTI 稳定在 5–6s。  
    当前实现是先请求 `/api/orders?withItems=true` 再在前端做聚合，接口平均耗时约 3s。  
    期望：
    1）首屏 TTI 控制在 2s 内；  
    2）不要影响现有筛选功能。  
    约束：后端暂时无法加缓存，只能在现有数据库和接口之上优化。"

两段话喂给 AI，输出质量会完全不一样；  
同样，两段话发给同事，对方的理解和行动也会完全不同。

---

总结一下这一章的核心：

- 问题描述说不清，99% 后面的智能都是"瞎忙活"；  
- 高质量的问题描述，至少要讲清楚对象、目标、约束和语境；  
- 尽量用行业内真实的专业语言，而不是"感觉化"的形容词；  
- 把问题描述写成"人和 AI 都能执行的任务说明书"，才算真正完成了从"感觉"到"可执行"的那一步。

## 6. "文本即代码"：从原理到一个落地案例

你可以把"文本即代码"理解成三层：

1. 文本是一种 **约定好的配置 / DSL**（领域语言）。
2. 程序负责 **解析这段文本**，把它变成结构化的任务。
3. 模型和工具负责 **根据这个任务去执行**（写 SQL、跑脚本、调接口）。

### 6.1 第一性原理：为什么文本可以当代码用？

从底层看，本质是两件事：

- 对人来说，自然语言比 JSON、YAML 更容易写，但如果加一点点结构（标题、列表、字段名），就能变成"人和机器都看得懂的说明书"。
- 对模型来说，只要格式稳定、语义清晰，它就能把这份说明书当成输入，输出一段可执行的东西（SQL、脚本、API 调用计划）。

所以，"文本即代码"其实是：

> 把"写配置 / 写代码"这件事，前移到"写一份更专业、更结构化的自然语言说明书"上。

### 6.2 一个完整案例：用 Markdown 任务卡驱动数据分析

假设你想做一套"BI 同学写 Markdown，系统自动帮他查数"的东西，流程可以是：

1. BI 同学写一张任务卡（Markdown 文件）：

```markdown
## 任务：统计本周新用户下单转化

- 背景：运营想看本周新注册用户的下单情况，用来评估拉新活动效果。
- 数据来源：数据仓库 `dw`，用户表 `dw_users`，订单表 `dw_orders`。
- 口径约束：
  - 新用户定义：在本周注册（`register_date` 在本周内）。
  - 下单定义：有至少一笔支付成功订单（`order_status = 'paid'`）。
- 输出要求：
  - 维度：按注册日期分组。
  - 指标：新注册人数、新用户下单人数、下单率（下单人数 / 注册人数）。
  - 时间范围：本周（含今天）。
```

2. 后端脚本做的事情非常"术"：

- 读取这份 Markdown，只关心几块固定结构的内容（背景、数据来源、口径约束、输出要求）。
- 把这些内容整理成一个结构化的对象，例如：

```json
{
  "datasource": "dw",
  "tables": ["dw_users", "dw_orders"],
  "metrics": [
    "new_users",
    "new_users_with_orders",
    "conversion_rate"
  ],
  "dimensions": ["register_date"],
  "filters": [
    "register_date in current_week",
    "order_status = 'paid'"
  ]
}
```

- 把这个对象 + 原始 Markdown 一起，作为上下文喂给模型，让它"只做一件事"：  
  根据这些信息生成一条符合公司规范的 SQL。

3. 模型的输出就是"代码"：

```sql
SELECT
  u.register_date,
  COUNT(DISTINCT u.user_id) AS new_users,
  COUNT(DISTINCT o.user_id) AS new_users_with_orders,
  COUNT(DISTINCT o.user_id) * 1.0 / COUNT(DISTINCT u.user_id) AS conversion_rate
FROM dw_users u
LEFT JOIN dw_orders o
  ON u.user_id = o.user_id
  AND o.order_status = 'paid'
WHERE u.register_date BETWEEN date_trunc('week', CURRENT_DATE)
                          AND CURRENT_DATE
GROUP BY u.register_date
ORDER BY u.register_date;
```

4. 程序拿到 SQL 后，去实际的数据仓库执行，并把结果表格返回给 BI 同学。

这里"文本即代码"的关键在于：

- BI 写的是一张结构化的"任务说明书"，里面用的是尽量专业的业务语言；
- 系统把这张说明书变成结构化任务，再让模型负责"翻译成 SQL"；
- 对 BI 来说，他不需要直接写 SQL；对系统来说，这张 Markdown 就是"高层次的源码"。

### 6.3 小结：术层面需要做的几件事

如果你想在自己的系统里实现类似能力，可以从下面几个动作入手：

- 先定义好"任务说明"的固定结构（哪些标题、哪些字段是必填的）；
- 在前端或文档层面，引导大家按照这个结构来写，而不是随便一大段话；
- 在后端写好"解析 + 清洗"逻辑，把这段文本变成结构化任务；
- 再把"结构化任务 + 原始文本"作为上下文喂给模型，让它只负责一件事：生成可执行的代码或操作计划；
- 最后，给这套链路加上最基本的校验和回滚机制，避免模型生成的东西直接"误伤生产"。

做到这一步，你写下的每一段"描述清楚的问题和任务"，就不只是沟通材料，而是真正意义上的"高层代码"。
